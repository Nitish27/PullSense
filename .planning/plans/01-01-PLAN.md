# Plan: Phase 1 GitHub App Skeleton Slice

**Goal:** Add the first real Phase 1 backend slice for PullSense: verified webhook intake, review job enqueueing, and a worker boundary that can fetch PR files through the GitHub package.
**Phase:** 1
**Requires:** `.planning/plans/00-01-PLAN.md`
**Provides:** Secure `/webhook` intake, queue payload shape, and a testable PR file retrieval workflow
**Context budget:** 3 tasks, ~75 minutes total

---

### Task 1: Webhook Signature Verification And Event Parsing

**Files:**
- Modify: `apps/api/package.json`
- Modify: `apps/api/src/app.test.ts`
- Modify: `apps/api/src/app.ts`
- Modify: `apps/api/src/env.ts`
- Create: `apps/api/src/github/github-webhook.ts`
- Create: `apps/api/src/github/github-webhook.test.ts`
- Create: `packages/shared/src/github.ts`
- Modify: `packages/shared/src/index.ts`

**Action:**
Add the secure webhook verification and event-parsing primitives needed by the API.

Implement:
- GitHub webhook env parsing for `GITHUB_WEBHOOK_SECRET` plus Redis/App credentials needed by the next task
- a small shared type module for the subset of GitHub pull request webhook data we need in Phase 1
- a pure utility that:
  - computes GitHub `sha256` webhook signatures
  - verifies the `x-hub-signature-256` header against a raw request body
  - normalizes the webhook payload into a review-job-ready shape only for supported `pull_request` actions
- tests for:
  - valid signature
  - invalid signature
  - supported `pull_request` event normalization
  - unsupported event/action filtering

DO NOT:
- fetch PR files yet
- enqueue jobs yet
- couple signature logic to Fastify internals

**Verify:**
```bash
pnpm --filter @ai-code-review/api test -- --config vitest.config.ts src/github/github-webhook.test.ts
# Expected: PASS with all webhook verification tests green
```

**Done when:**
- [ ] Raw-body signature verification is implemented in a pure testable utility
- [ ] Shared Phase 1 webhook payload types exist
- [ ] Only supported `pull_request` actions normalize into review jobs

### Task 2: Webhook Route And Queue Enqueue Flow

**Files:**
- Modify: `apps/api/package.json`
- Modify: `apps/api/src/app.test.ts`
- Modify: `apps/api/src/app.ts`
- Create: `apps/api/src/queue/review-queue.ts`
- Create: `apps/api/src/routes/webhook.ts`
- Create: `apps/api/src/routes/webhook.test.ts`

**Action:**
Wire the verified webhook flow into Fastify and enqueue review jobs.

Implement:
- Fastify raw-body support so the webhook route can verify GitHub signatures against the exact request bytes
- a queue abstraction with:
  - a small interface the app can depend on in tests
  - a BullMQ-backed implementation for runtime
- `/webhook` handling that:
  - accepts GitHub webhook headers
  - verifies the signature
  - filters to `pull_request` events/actions
  - enqueues a normalized review job payload
  - returns a clear status for ignored vs accepted events
- tests for:
  - accepted valid pull request webhook enqueues a job
  - invalid signature returns 401
  - unsupported event returns an ignored/accepted-noop response without queueing

DO NOT:
- connect directly to Redis in route tests
- fetch PR files inside the webhook request lifecycle
- add non-Phase-1 GitHub comment or AI review behavior

**Verify:**
```bash
pnpm --filter @ai-code-review/api test -- --config vitest.config.ts src/routes/webhook.test.ts
# Expected: PASS with enqueue/401/ignored route tests green
```

**Done when:**
- [ ] `/webhook` exists and verifies GitHub signatures using raw body
- [ ] Valid supported PR events enqueue a review job
- [ ] Invalid signatures are rejected
- [ ] Unsupported events/actions do not enqueue work

### Task 3: Worker Boundary And PR File Fetching

**Files:**
- Modify: `packages/github/package.json`
- Modify: `packages/github/src/index.ts`
- Create: `packages/github/src/pull-request-files.ts`
- Create: `packages/github/src/pull-request-files.test.ts`
- Modify: `apps/api/src/env.ts`
- Modify: `apps/api/src/queue/review-queue.ts`
- Create: `apps/api/src/workers/review-worker.ts`
- Create: `apps/api/src/workers/review-worker.test.ts`

**Action:**
Add the first worker-side GitHub integration boundary for retrieving changed PR files.

Implement:
- a GitHub package helper that can:
  - create an authenticated app/installation client from env-driven credentials
  - fetch PR files from `GET /repos/{owner}/{repo}/pulls/{pull_number}/files`
- a worker handler that consumes the normalized review job payload and calls the GitHub helper
- worker-side logging that proves the diff-fetch path works without introducing actual review generation yet
- tests that mock the GitHub package boundary and verify:
  - the handler requests PR files with the right owner/repo/number/install context
  - the returned file list is surfaced in a structured result

DO NOT:
- post comments yet
- call AI models yet
- mix indexing concerns into the worker

**Verify:**
```bash
pnpm --filter @ai-code-review/api test -- --config vitest.config.ts src/workers/review-worker.test.ts
# Expected: PASS with worker fetch tests green

pnpm --filter @ai-code-review/github test
# Expected: PASS with GitHub package tests green

pnpm lint
pnpm typecheck
pnpm test
pnpm build
# Expected: all pass
```

**Done when:**
- [ ] GitHub package exposes a PR file retrieval boundary
- [ ] Worker handler consumes review jobs and calls that boundary correctly
- [ ] Full workspace verification passes after the Phase 1 slice
